كيف تمنع تكرار التنبيهات في أنظمة الأتمتة؟

لا يصبح تدفق التنبيهات موثوقًا لمجرد أنه يرسل رسالة. المطلوب أن يرسل الرسالة الصحيحة إلى المستلم الصحيح مرة واحدة لكل حدث تشغيلي، حتى عند تداخل المشغلات أو انقطاع الاتصال أو إعادة تنفيذ التدفق.

المبدأ التشغيلي: إعادة المحاولة يجب أن تستعيد التنفيذ الفاشل، لا أن تنشئ إجراءً تجاريًا ثانيًا. لذلك يحتاج النظام إلى سجل دائم يحدد الحدث الذي عولج والتنبيه الذي أرسل فعليًا.

1. لماذا تتكرر التنبيهات؟

غالبًا لا تكون المشكلة في البريد أو تيليجرام، بل في تصميم التنفيذ. من الأسباب الشائعة: تشغيل Triggerين في وقت متقارب، تحديث المستخدم لصفحة تستدعي منطق التنبيه، انتهاء مهلة الاتصال بعد أن قبل مزود الخدمة الرسالة، أو إعادة المنصة للمحاولة بعد فشل إحدى الخطوات.

الحالة الأخطر هي الفشل الجزئي: يرسل المزود الرسالة بالفعل، لكن التدفق يفشل قبل تسجيل النجاح. عند التشغيل التالي لا يجد النظام دليلًا على الإرسال فيكرر الرسالة. خانة واحدة باسم «تم الإرسال» أو «وقت آخر تشغيل» لا تكفي، لأن التشغيل الواحد قد يشمل عشرات الموظفين والمستندات والمستلمين.

2. عرّف الحدث التشغيلي قبل صياغة الرسالة

يجب التمييز بين واقعة التنفيذ والحدث التجاري. عبارة «اشتغل السكربت الساعة الثامنة» تصف تنفيذًا تقنيًا، بينما «دخل المستند رقم 184 لأول مرة في نطاق التنبيه قبل 30 يومًا» تصف حدثًا تشغيليًا.

أنشئ مفتاحًا ثابتًا من العناصر التي تحدد فرادة الحدث:

event_key = SHA256(
  event_type + "|" + entity_id + "|" + threshold + "|" + effective_date
)

مثال:
document_expiry|DOC-184|30-days|2026-08-11

يجب أن يظل المفتاح نفسه في كل إعادة محاولة. لا تضف الوقت الحالي أو رقم التنفيذ أو قيمة عشوائية، لأنها تجعل كل محاولة تبدو كحدث جديد.

3. استخدم سجل تنبيهات مستقلًا

يسجل النظام صفًا واحدًا لكل تركيبة من الحدث والمستلم والقناة. هذا أفضل من إضافة أعمدة مثل «تم إرسال البريد» داخل سجل الموظف، لأن الحدث الواحد قد يذهب إلى عدة أشخاص وبمستويات تصعيد وقنوات مختلفة.

الحقلالغرض
event_keyالمعرف الثابت للحدث.
recipientالبريد أو Chat ID أو وجهة الإرسال.
channelالبريد أو تيليجرام أو أي مزود آخر.
statusقيد الانتظار، جارٍ الإرسال، تم، فشل قابل للمحاولة، فشل دائم.
attempt_countعدد محاولات التسليم.
provider_referenceرقم الرسالة أو مرجع المزود إن توفر.
sent_atوقت تأكيد طلب الإرسال.
last_errorآخر خطأ تشخيصي دون أسرار أو بيانات حساسة.

يجب منع وجود سجلين نشطين لنفس event_key + recipient + channel باستخدام قيد فريد في قاعدة البيانات أو منطق مكافئ له.

4. تعامل مع الإرسال كآلة حالات

الخانة المنطقية نعم/لا لا تمثل حالات عدم اليقين. استخدم حالات واضحة:

PENDING → SENDING → SENT ↘ RETRYABLE_FAILURE → SENDING ↘ PERMANENT_FAILURE
  • قيد الانتظار: الحدث موجود ولم يبدأ الإرسال.
  • جارٍ الإرسال: عملية معينة حجزت السجل. أضف وقت انتهاء للحجز حتى يمكن استعادة العمل المتروك.
  • تم الإرسال: اكتملت استجابة المزود وسجلت النتيجة.
  • فشل قابل للمحاولة: مهلة أو Rate Limit أو عطل مؤقت.
  • فشل دائم: عنوان أو صلاحية أو بيانات اعتماد أو Payload غير صالح.

حدد عددًا أقصى للمحاولات وتأخيرًا متزايدًا بينها. يجب أن تكون القيم إعدادات قابلة للتغيير وفق أهمية العملية وحدود مزود الخدمة.

5. امنع عمليتين من حجز الحدث نفسه

قبل الإرسال احصل على Lock أو نفذ حجزًا ذريًا للسجل. في Google Apps Script يمكن استخدام LockService لحماية الجزء الحرج من التنفيذ. وفي قواعد البيانات حدّث السجل فقط إذا كانت حالته تسمح بالحجز ثم تحقق أن صفًا واحدًا فقط تأثر.

claim = update notification_ledger
        set status = 'SENDING', lease_until = now + 5 minutes
        where event_key = ?
          and status in ('PENDING', 'RETRYABLE_FAILURE')
          and next_attempt_at <= now

if affected_rows != 1:
    stop; another execution owns the event

الـLock يحمي لحظة التنافس، لكنه لا يغني عن السجل الدائم؛ لأن الأقفال تنتهي والعمليات قد تتوقف والأنظمة يعاد تشغيلها.

6. تسلسل التنفيذ الصحيح

  1. اقرأ سجل المصدر وتحقق من صحته.
  2. احسب مفتاح الحدث الثابت.
  3. أنشئ سجل التنبيه إن لم يكن موجودًا.
  4. احجز سجلًا مؤهلًا بطريقة ذرية.
  5. كوّن الرسالة من Template له رقم إصدار.
  6. استدعِ مزود الخدمة بمهلة زمنية محددة.
  7. سجل الاستجابة والمرجع ثم غيّر الحالة إلى SENT.
  8. عند الفشل صنف الخطأ وحدد محاولة لاحقة منضبطة.

إذا كان المزود يدعم Idempotency Key فأرسل إليه مفتاح الحدث نفسه في جميع المحاولات. هذا يقلل احتمال التكرار عندما يقبل المزود الرسالة بينما يفقد العميل الاستجابة.

7. نمط عملي في Google Apps Script

في الأنظمة الداخلية الصغيرة يمكن استخدام Sheet مخصص كسجل تنبيهات. يناسب PropertiesService الإعدادات الصغيرة ومؤشرات الاستمرار، لكنه لا ينبغي أن يتحول إلى بديل غير منظم عن سجل قابل للبحث والمراجعة.

function processAlerts() {
  const lock = LockService.getScriptLock();
  lock.waitLock(30000);
  try {
    const events = findEligibleBusinessEvents();
    events.forEach(event => {
      const key = buildStableEventKey(event);
      const record = getOrCreateLedgerRecord(key, event.recipient, 'email');
      if (!claimRecord(record)) return;
      deliverAndRecord(record, event);
    });
  } finally {
    lock.releaseLock();
  }
}

اجعل دوال الـTrigger قصيرة. فتح الصفحة أو تعديل خلية أو استقبال Web Request يجب أن يقيّم العمل أو يضعه في طابور، لا أن يشغل عشوائيًا دورة التنبيهات كاملة.

8. نمط عملي في n8n

ضع خطوة البحث في سجل التنبيهات وحجز الحدث قبل عقدة البريد أو تيليجرام. وجّه الأخطاء المؤقتة إلى مسار إعادة محاولة، والأخطاء الدائمة إلى قائمة مراجعة تشغيلية. استخدم Error Workflow منفصلًا للإبلاغ عن فشل التنفيذ، مع منع تنبيه الخطأ من إعادة تشغيل التدفق الأساسي.

عند ارتفاع الحجم استخدم قاعدة بيانات بمفتاح فريد وحجز Transactional بدل الاعتماد على Static Data داخل التدفق. سجل تنفيذ n8n مفيد للتشخيص، لكنه ليس السجل التجاري للحدث وقد يخضع للحذف وفق إعدادات الاحتفاظ.

9. قِس الموثوقية لا عدد الرسائل

المؤشرماذا يعني؟
معدل التكرارعدد الرسائل المكررة ÷ الأحداث التي نجحت. الهدف العملي أن يقترب من الصفر.
نسبة التسليم في الوقتالأحداث التي وصلت إلى SENT داخل المهلة المطلوبة.
نسبة التعافي بالمحاولةالأخطاء المؤقتة التي اكتملت دون تدخل يدوي.
عمر الفشل الدائمالمدة التي تظل فيها العناوين أو الصلاحيات غير الصالحة دون حل.
نسبة التنبيه إلى الإجراءهل أنتج التنبيه فعلًا تشغيليًا أم أضاف ضوضاء فقط؟

كثرة الرسائل ليست دليلًا على نجاح الأتمتة. النظام الناضج يقلل التكرار ويظهر الاستثناءات ويحدد التنبيهات التي أدت إلى إجراء حقيقي.

10. الحد الأدنى من ضوابط الإنتاج

  • تحقق من المستلم ومحتوى الرسالة قبل حجز الإرسال.
  • لا تسجل كلمات المرور أو Tokens أو بيانات الموظفين الحساسة.
  • افصل مستلمي الاختبار عن الإنتاج.
  • امنح كل Template رقم إصدار وسجل الإصدار المستخدم.
  • أنشئ وظيفة إعادة إرسال يدوية منضبطة تسجل Override مصرحًا أو حدثًا جديدًا واضحًا.
  • أنشئ تقرير مطابقة بين الأحداث المؤهلة التي لم ترسل والسجلات المرسلة التي لا يقابلها حدث صالح.
  • حدد سياسة احتفاظ لبيانات المصدر ولسجل التنبيهات.
الأتمتة الموثوقة ليست التي لا تفشل أبدًا، بل التي تفشل بصورة ظاهرة، وتعيد المحاولة بأمان، وتحتفظ بأدلة تفسر كل إجراء تشغيلي.
