How to Prevent Duplicate Alerts in Business Automation

A notification workflow is not reliable merely because it sends a message. It must send the correct message, to the correct recipient, once per business event—even when triggers overlap, APIs time out or the workflow is retried.

Published: 12 July 202612 min readHossam Ali
Use caseEmail, Telegram and workflow alerts
LevelIntermediate to advanced
Core principleSafe retries through idempotency

Operating principle: a retry should recover a failed execution, not create a second business action. The workflow therefore needs a durable record of what event was processed and what notification was actually delivered.

1. Why duplicate notifications happen

Duplicate messages usually come from execution design, not from the messaging channel. Common causes include two scheduled triggers running close together, a user refreshing a web app that also invokes notification logic, a timeout after the provider accepted the message, or a workflow platform automatically retrying a failed step.

The difficult scenario is a partial failure: the external service sends the email or Telegram message, but the workflow fails before recording success. The next run sees no success record and sends the message again. A simple “last run timestamp” does not solve this because one run may contain many employees, documents or recipients.

2. Define the business event before the message

The system must identify the event independently from the execution. “The script ran at 08:00” is an execution fact. “Employee document 184 entered the 30-day warning window for the first time” is a business event.

Create a stable event key from fields that define uniqueness:

event_key = SHA256(
  event_type + "|" + entity_id + "|" + threshold + "|" + effective_date
)

Example:
document_expiry|DOC-184|30-days|2026-08-11

The key must remain identical across retries. Do not include the current timestamp, execution ID or random number. Those values make every retry look like a new event.

3. Use a notification ledger, not a single status cell

A ledger stores one row per event-recipient-channel combination. This is more reliable than adding columns such as “Email Sent” to the source record because the same event may have several recipients, escalation levels and delivery channels.

FieldPurpose
event_keyStable identifier for the business event.
recipientEmail address, chat ID or destination identifier.
channelEmail, Telegram, WhatsApp or another provider.
statusPending, sending, sent, retryable-failure or permanent-failure.
attempt_countNumber of delivery attempts.
provider_referenceMessage ID or response reference when available.
sent_atConfirmed delivery-request timestamp.
last_errorLatest diagnostic information without secrets.

Use a unique constraint—or its logical equivalent—on event_key + recipient + channel. The workflow should refuse to create a second active record for the same combination.

4. Treat delivery as a state machine

A Boolean field cannot represent uncertainty. The workflow needs explicit states:

PENDING → SENDING → SENT ↘ RETRYABLE_FAILURE → SENDING ↘ PERMANENT_FAILURE
  • Pending: the event exists but delivery has not started.
  • Sending: a worker has claimed the record. Store a lease timestamp so abandoned work can be recovered.
  • Sent: the provider call completed successfully and the result was recorded.
  • Retryable failure: a temporary timeout, rate limit or service error occurred.
  • Permanent failure: the address, credentials, payload or permission is invalid and needs intervention.

Retries should be limited and delayed. A typical pattern is exponential backoff with a maximum attempt count. The exact values depend on operational urgency and provider limits; they should be configurable rather than hard-coded.

5. Prevent concurrent workers from claiming the same event

Before sending, acquire a lock or perform an atomic claim. In Google Apps Script, LockService can prevent overlapping executions from entering the same critical section. In a database-backed workflow, update the row only when its current status is PENDING or an expired SENDING lease, then verify that exactly one row was claimed.

claim = update notification_ledger
        set status = 'SENDING', lease_until = now + 5 minutes
        where event_key = ?
          and status in ('PENDING', 'RETRYABLE_FAILURE')
          and next_attempt_at <= now

if affected_rows != 1:
    stop; another execution owns the event

A lock protects the critical section. It does not replace the durable ledger: locks expire, processes crash and systems restart.

6. Use the correct processing sequence

  1. Read and validate the source record.
  2. Calculate the stable event key.
  3. Insert the ledger record if it does not already exist.
  4. Atomically claim an eligible ledger record.
  5. Build the message from a versioned template.
  6. Call the provider with a timeout.
  7. Record the provider response and mark the row SENT.
  8. If the call fails, classify the error and schedule a controlled retry.

Where a provider supports its own idempotency key, send the same event key with every retry. This reduces the remaining ambiguity when the provider accepts a message but the client loses the response.

7. Practical pattern for Google Apps Script

For small internal systems, use a dedicated sheet or datastore as the ledger. PropertiesService is suitable for compact configuration and cursors, but it should not become an unstructured replacement for a queryable notification history.

function processAlerts() {
  const lock = LockService.getScriptLock();
  lock.waitLock(30000);
  try {
    const events = findEligibleBusinessEvents();
    events.forEach(event => {
      const key = buildStableEventKey(event);
      const record = getOrCreateLedgerRecord(key, event.recipient, 'email');
      if (!claimRecord(record)) return;
      deliverAndRecord(record, event);
    });
  } finally {
    lock.releaseLock();
  }
}

Keep trigger handlers thin. A page load, edit event or web request should enqueue or evaluate work; it should not casually execute the full notification routine.

8. Practical pattern for n8n

In n8n, place the ledger lookup and claim before the email or Telegram node. Route temporary errors into a retry path and permanent errors into an operational review queue. Use a separate error workflow to report failed executions, but do not let the error notification itself recursively trigger the main workflow.

For higher volume, use a database with a unique key and transactional claim rather than workflow static data. Execution history is useful for diagnosis, but it is not the business ledger and may be pruned according to retention settings.

9. Measure reliability, not just message count

MetricInterpretation
Duplicate rateDuplicate deliveries divided by successful business events. Target should be effectively zero.
Delivery success rateEvents reaching SENT within the required time window.
Retry recovery rateTemporary failures later completed without manual action.
Permanent failure ageHow long invalid destinations or permissions remain unresolved.
Alert-to-action ratioWhether notifications produce a real operational response or merely create noise.

High message volume is not evidence of effective automation. A mature system reduces duplicates, identifies exceptions and shows which alerts resulted in action.

10. Minimum production controls

  • Validate recipient and payload before claiming delivery.
  • Do not log access tokens, passwords or sensitive employee data.
  • Separate test and production recipients.
  • Version message templates and record the version used.
  • Provide a controlled manual resend action that creates an explicit new event or authorized override.
  • Run a reconciliation report: eligible events without sent records, and sent records without valid source events.
  • Define retention rules for both the source data and notification ledger.
A reliable automation is not one that never fails. It is one that fails visibly, retries safely and preserves enough evidence to explain every business action.
